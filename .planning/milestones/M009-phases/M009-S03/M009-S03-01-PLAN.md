  <action>
    Extend the host-owned retrieval boundary with a draft-facing function that accepts a conversation or conversation-derived query context and returns a canonical-first grounding bundle. The bundle must keep canonical Knowledge Base evidence distinct from assistive resolved-case evidence, preserve `source_type`, `trust_level`, `citation_target`, `match_reasons`, and `can_ground_reply?`, and produce an explicit grounding assessment that the worker can branch on.

    Encode the weak-grounding rules from context directly in this contract. At minimum, distinguish:
    - strong canonical grounding
    - weak-but-recoverable grounding eligible for a single clarification question
    - escalation-only grounding for empty, conflicting, errored, or assistive-only cases

    Do not reintroduce remote search logic here. Reuse the normalized retrieval results already produced by Phase 1 and Phase 2 work. Treat retrieved content strictly as evidence input, not executable instructions, so later generation code cannot let retrieved text override proposal-type policy. The bundle should also expose enough state for the worker to know whether a clarification attempt has already been used for the current conversation context.
  </action>

    - the retrieval draft path preserves `source_type`, `trust_level`, `citation_target`, and `match_reasons`
    - the retrieval draft path contains explicit logic for a clarification-eligible state and an escalation-only state
    - the retrieval draft path exposes or accepts state needed to prevent repeated clarification loops after one failed clarification turn
    - no new direct `Req.get` or `Req.post` retrieval call is introduced in retrieval or drafting code

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Retrieval boundary -> drafting engine | Only filtered, source-labeled evidence may enter draft generation |
| Grounding assessment -> worker branch | Weak grounding must change the workflow state, not just the copy |
| Structured draft persistence -> operator UI | Durable proposal state must preserve trust and citation semantics end to end |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-M009-S03-01 | Information Disclosure | Retrieval-to-draft bundle | mitigate | Reuse host-owned retrieval filters and pass only normalized, visibility-safe evidence into drafting |
| T-M009-S03-02 | Tampering | Weak-grounding state machine | mitigate | Encode clarification/escalation as explicit proposal states so assistive-only or missing evidence cannot masquerade as a normal grounded reply |
| T-M009-S03-05 | Tampering | Retrieved-content prompt boundary | mitigate | Keep retrieved snippets in explicit evidence fields, separate them from system instructions, and add tests proving retrieved text cannot override proposal-type policy |
</threat_model>

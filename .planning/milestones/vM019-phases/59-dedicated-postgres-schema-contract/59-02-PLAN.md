<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| migration source -> Postgres catalogs | Ecto migration code creates database objects in host-owned databases. |
| raw SQL -> Postgres parser | Migration SQL strings bypass Ecto object qualification. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59-06 | Tampering / Elevation of Privilege | migration prefix interpolation | mitigate | Use `Cairnloop.SchemaPrefix.quote_identifier!/1` and `quoted_table/2`; never interpolate raw config. |
| T-59-07 | Tampering | migration DDL helpers | mitigate | Add `prefix: prefix` to tables, indexes, references, and drops instead of relying on search path. |
| T-59-08 | Information Disclosure / Integrity | public/dedicated split | mitigate | Catalog tests prove objects are physically in the configured schema. |
| T-59-09 | Denial of Service | vector extension rollback | mitigate | Source-scan and migration source prohibit `DROP EXTENSION vector`. |
| T-59-10 | Denial of Service / Integrity | Oban redirection | mitigate | Do not add Oban references to library migrations; Oban remains host-owned. |
</threat_model>

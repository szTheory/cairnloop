# Phase 58 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase is a trust-boundary hardening phase, not a new product-surface phase. UI work is limited to operator-facing and adopter-facing trust state that implementation touches: identity separation, ingress failure states, Scrypath opt-in state, doctor/readiness output, audit/troubleshooting surfaces, Settings health rows, and conversation evidence rail warnings.

Primary sources: `58-CONTEXT.md` decisions D-01 through D-15, `58-RESEARCH.md` Standard Stack and Architecture Patterns, `58-VALIDATION.md`, `CLAUDE.md`, `docs/operator-ui-principles.md`, `prompts/cairnloop_brand_book.md`, `priv/static/cairnloop.css`, and `lib/cairnloop/web/components.ex`.

## Design System

| Property | Value |
|----------|-------|
| Tool | Existing Cairnloop `.cl-*` / BEM token system. No Tailwind. No shadcn. No registry blocks. |
| Preset | Not applicable. This is Phoenix LiveView with Cairnloop's shipped CSS, not React/Next/Vite. |
| Component library | `Cairnloop.Web.Components` plus `priv/static/cairnloop.css`; use `cl_button`, `cl_card`, `cl_chip`, `cl_banner`, `cl_flash`, `cl_empty`, `cl_disclosure`, `cl_fact_list`, `cl_source_card`, `cl_status_cell`, `cl_switch`, `cl_page`, and `cl_shell` before adding new markup. |
| Icon library | Built-in inline `cl_icon` SVG set. Add icons only through `Cairnloop.Web.Components`; no external icon package. |
| Font | `--cl-font-sans` for UI, `--cl-font-display` only for existing hero/count accents, `--cl-font-mono` only for code, IDs, trace labels, and disclosure-only technical detail. |

### System Contract

- Keep adopter-facing dashboard work inside the shared component system. If a trust-state pattern repeats on Settings, Audit, Conversation, or docs-rendered examples, add or extend a `.cl-*` component/class rather than using one-off inline styles.
- `priv/static/cairnloop.css` remains the visual source of truth. New spacing, motion, radius, color, shadow, z-index, and control sizing must read from `--cl-*` tokens.
- Do not introduce Tailwind classes, shadcn components, package CSS, JS UI dependencies, or third-party registry blocks.
- Operator copy must be calm, fail-closed, reason-forward, and honest. Do not show raw Elixir terms, `inspect/1` output, raw JSON, support message bodies, secrets, or request payloads in default views.
- Raw technical detail is allowed only behind an explicit native disclosure (`cl_disclosure` / `.cl-details`) with a human summary outside the disclosure.

### Surface Contracts

| Surface | Contract |
|---------|----------|
| Settings / System health | If trust checks are surfaced in dashboard Settings, place them in the existing health pattern using `cl_card`, `cl_fact_list`, and `cl_chip`. Group rows as "Ingress", "MCP authentication", "Telemetry and logs", and "Optional automation" only if implementation adds enough checks to justify grouping. |
| Conversation evidence rail | If identity or side-effect state affects a conversation action, show a persistent `cl_banner` before the affected action group. Do not move the message timeline. Do not collapse Tier 1 safety content. Raw snapshots remain inside `cl_disclosure`. |
| Audit Log | Actor, action, reason, and details must stay humanized through presenter functions. Missing operator attribution for an operator action must read as a wiring problem, not as a customer or guessed system actor. Metadata remains behind "View details". |
| Doctor / readiness output | Treat `mix cairnloop.doctor` as the primary readiness/trust diagnostic surface. Output should name the state, why it matters, and the next host action. `/health` copy stays shallow liveness only. |
| Troubleshooting docs | Add only phase-relevant trust and readiness copy. Use practical host-owned language, failure modes, and verification steps. Do not broaden Phase 60 docs scope. |

## Spacing Scale

Declared values use the existing 4px rhythm and `--cl-space-*` tokens:

| Token | Value | Usage |
|-------|-------|-------|
| xs | `var(--cl-space-2)` = 4px | Icon gaps, field label gaps, compact disclosure internals |
| sm | `var(--cl-space-3)` = 8px | Inline control gaps, chip/icon spacing, compact stack spacing |
| md | `var(--cl-space-5)` = 16px | Default card body padding, banner padding, normal vertical groups |
| lg | `var(--cl-space-7)` = 24px | Page section separation, card-to-card spacing, conversation rail gap |
| xl | `var(--cl-space-8)` = 32px | Desktop page padding and major layout gaps |
| 2xl | `var(--cl-space-10)` = 48px | Empty state vertical padding and rare large breaks |
| 3xl | `var(--cl-space-11)` = 64px | Page-level spacing only; avoid in dense operator panels |

Exceptions:

- `var(--cl-space-1)` = 2px exists for optical fine-tuning in current CSS. Do not use it for new phase layout spacing.
- Touch targets for icon-only controls, checkboxes, switches, and compact rail controls must be at least 44px using `--cl-control-h-lg`, `.cl-checkbox`, `.cl-switch`, or equivalent shared CSS.
- Conversation evidence rail desktop width stays `--cl-rail-width` / 352px. Mobile stacks vertically; do not create a second rail width token.
- Breakpoint conditions stay literal `640px`, `768px`, and `1024px` as already documented in CSS. Do not put `var()` inside `@media` conditions.

## Typography

Use the existing product UI scale; do not introduce additional sizes for this phase.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | `var(--cl-font-body)` = 15px | `var(--cl-weight-regular)` = 400 | `var(--cl-leading-body)` = 24px |
| Label | `var(--cl-font-small)` = 13px | `var(--cl-weight-semibold)` = 600 | `var(--cl-leading-small)` = 20px |
| Heading | `var(--cl-font-panel)` = 18px | `var(--cl-weight-semibold)` = 600 | `var(--cl-leading-panel)` = 26px |
| Display | `var(--cl-font-title)` = 28px | `var(--cl-weight-semibold)` = 600 | `var(--cl-leading-title)` = 36px |

Typography rules:

- Use only two semantic weights in new UI work: regular 400 and semibold 600. Existing shared components that already use `--cl-weight-medium` may remain unchanged; do not introduce it into new phase-specific markup.
- Use `--cl-font-mono` only for copied tokens, command names, opaque IDs, and trace values. Technical detail must be disclosure-only unless the user needs to copy it immediately.
- Do not use raw code identifiers as operator labels. Use "Operator identity", "Customer reference", "Widget verifier", "MCP token", "Scrypath automation", and "Diagnostic mode" in UI copy.

## Color

Use Cairnloop semantic tokens only. Do not hardcode hex in render code or new CSS except token fallback values inside `priv/static/cairnloop.css`.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60% minimum; Cairnloop dashboard target 70%) | `var(--cl-bg)`, `var(--cl-surface)`, `var(--cl-surface-raised)`, `var(--cl-surface-sunken)` | Page background, cards, Settings/Audit/Conversation panels, disclosure bodies |
| Secondary (30% maximum; Cairnloop dashboard target 20%) | `var(--cl-text)`, `var(--cl-text-muted)`, `var(--cl-border)`, `var(--cl-border-strong)` | Primary text, secondary copy, dividers, table borders, inactive structure |
| Accent (10%) | `var(--cl-primary)` / Path Copper | Primary CTA, active route marker, current nav underline, focus ring, selected/current state marker |
| Destructive | `var(--cl-danger)`, `var(--cl-danger-surface)`, `var(--cl-danger-border)`, `var(--cl-danger-text)` | Blocked auth, failed trust checks, revoked-token actions, destructive confirmations only |

Accent reserved for: primary action buttons, active route markers, current nav state, selected/current status marker, and visible focus. Do not use copper for whole cards, full-width warnings, decorative fills, or every interactive element.

Semantic state mapping:

| State | Component Variant | Copy Meaning |
|-------|-------------------|--------------|
| Configured / ready / verified | `success` | Safe to proceed; checked by this surface |
| Informational / not checked here | `info` or `neutral` | Honest status without implying readiness |
| Disabled by host choice | `neutral` or `warning` depending on risk | Inert by default; no side effect will run |
| Missing host seam / misconfigured | `danger` | Fail-closed; host action required |
| Diagnostic mode or opt-in side effect | `warning` | Host deliberately enabled a higher-risk mode |
| AI/eval/trace context | `ai` | Trace, draft, eval, and model-output context only |

State must never be color-only. Pair color with text and an icon via `cl_chip`, `cl_banner`, `cl_status_cell`, or `cl_source_card`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Review trust checks` |
| Empty state heading | `No trust checks need attention` |
| Empty state body | `Cairnloop did not find an ingress, identity, telemetry, or side-effect configuration that needs action on this surface.` |
| Error state | `Cairnloop blocked this action because operator identity is missing. Sign in through the host app and reload the dashboard before trying again.` |
| Destructive confirmation | `Revoke MCP token`: `Revoke token? Active integrations using it will fail immediately.` |

### Required Phase Copy Examples

Use these exact copy patterns, adjusted only for the specific surface name or host config label:

| Scenario | Default Copy |
|----------|--------------|
| Fail-closed operator auth | `Operator identity is missing. Cairnloop withheld this governed action so the audit trail does not guess who acted.` |
| Widget verifier missing | `Widget ingress is blocked because no host verifier is configured. Add a verifier before accepting customer sessions in production.` |
| Email webhook auth missing | `Email webhook ingress is blocked until the host configures request authentication. Cairnloop did not enqueue this payload.` |
| MCP token missing | `MCP request blocked. Provide a valid Bearer token before Cairnloop lists tools or accepts tool calls.` |
| Scrypath disabled | `Scrypath automation is disabled. Resolved conversations will stay inside Cairnloop unless the host opts in.` |
| Scrypath misconfigured | `Scrypath automation is enabled but not ready. Add a real API URL and key before Cairnloop can enqueue external indexing.` |
| Sensitive telemetry default | `Default telemetry is bounded. Cairnloop reports event names and outcomes, not customer message bodies, secrets, or raw payloads.` |
| Diagnostic mode enabled | `Diagnostic mode is enabled by host config. Use it only for a bounded investigation because additional support detail may be exported.` |
| `/health` liveness | `/health is liveness only. It confirms the app can answer HTTP; run doctor for database, Oban, notifier, retrieval, ingress, and automation checks.` |
| Doctor all clear | `Ready: Cairnloop checked the configured trust seams and did not find a blocking setup issue.` |
| Doctor blocked | `Blocked: Cairnloop found a trust configuration that can leak attribution, accept unsafe ingress, or enqueue an unchosen side effect. Fix the listed host setting before production use.` |
| Doctor not checked | `Not checked here: this surface cannot prove that dependency. Use the host app or integration test named below.` |
| Raw details disclosure summary | `Technical details` |
| Raw details disclosure body intro | `Use these details for debugging. Do not paste customer message bodies, secrets, or raw webhook payloads into shared channels.` |

### Copy Rules

- Say "operator" for dashboard users and "customer" for browser/widget users. Never use customer/session identity as the actor in operator copy.
- Use "customer reference" for `customer_ref` in UI. The raw field name belongs in docs or disclosure-only technical detail.
- Prefer "blocked", "withheld", "disabled", "not configured", and "not checked here" over vague "failed".
- Do not expose Elixir atoms, changesets, raw maps, raw JSON, request bodies, or `inspect/1` strings to operators. Presenter functions must humanize values first.
- Do not claim readiness from `/health`. Do not claim Scrypath, Oban, DB, pgvector, notifier, or MCP checks happened unless that surface actually performed them.

## Interaction Contracts

### Identity Separation

- Governed actions, recovery actions, resolve actions, approvals, search, and audit attribution must use the dashboard live session operator identity. They must not read actor identity from widget/customer/session fields.
- When operator identity is missing, mutation controls must fail closed. Use a persistent `cl_banner` with the fail-closed copy above and disable or withhold the unsafe action button. The conversation timeline may remain readable.
- If customer/session identity is displayed, label it "Customer reference" and treat it as opaque host-owned context. Do not style it as an actor, operator, account owner, or audit principal.

### Ingress Trust States

- Widget, email webhook, and MCP auth states must use the same visual vocabulary: `Configured`, `Blocked`, `Disabled`, `Not checked here`, or `Needs host config`.
- Missing widget/email/MCP auth seams are `danger` banners or chips because production behavior must fail closed.
- Public MCP OAuth/resource discovery may be documented as public discovery only; tool metadata and write surfaces must not be visually or verbally described as available before token validation.

### Side Effects

- Scrypath disabled is a calm neutral/warning state, not an error. The copy must make clear no external work is enqueued.
- Scrypath enabled with placeholder or missing credentials is a danger state. Do not show a "Run anyway" action.
- If a toggle is added for optional automation, use `cl_switch` with visible label text, `role="switch"`, correct `aria-checked`, and a reason-forward warning before enabling.

### Raw Technical Details

- Default view: humanized state, reason, and next action.
- Disclosure summary: short noun phrase such as `Technical details`, `Raw input snapshot`, or `Identifiers and trace`.
- Disclosure body: `.cl-code-block`, `cl_fact_list`, or existing trace style. Escape values. Never use `raw/1`.
- Do not put raw detail inside a tooltip; it must be keyboard reachable through native disclosure.

## Accessibility

- Use native controls first: links navigate, buttons mutate, `details` discloses, select/input controls keep explicit labels.
- Icon-only controls require accessible names. Prefer icon + text for trust and side-effect actions.
- Focus visibility must use `--cl-focus-ring`; do not remove outlines without the tokenized replacement.
- State is never color-only. Every blocked/ready/disabled state needs text plus an icon or shape.
- Tables with horizontal overflow must use `.cl-table-scroll` with `role="region"`, `tabindex="0"`, and an `aria-label`.
- Banners that block an action should use `role="status"` for non-urgent status and `role="alert"` only when immediate attention is required.
- Mutation controls affected by missing identity or auth should be disabled with visible explanatory copy nearby, not only hidden.
- All controls must remain keyboard reachable on Settings, Audit, Conversation evidence rail, and any optional readiness surface.

---

## Responsive Behavior

- Preserve cockpit scanability on desktop: wide pages remain `cl_page width="wide"` and conversation detail keeps timeline plus right evidence rail at the established 1024px breakpoint.
- On mobile, stack the conversation timeline and evidence rail vertically. Put trust-blocking banners before the action group they affect.
- Settings trust rows and doctor/readiness summaries must stack into one column below 640px and avoid horizontal scrolling unless rendering a table.
- Audit table remains inside `.cl-table-scroll`; do not force metadata details into the visible table cell by default.
- Long tokens, URLs, or IDs must wrap with the existing code block behavior. Do not let a token widen the viewport.
- Button text must fit at 320px width. Use concise labels such as `Review trust checks`, `Add MCP token`, `Revoke token`, `Load more`, and `Run doctor`.

---

## Motion

- No new motion is required for Phase 58. Prefer stable trust-state displays over animated emphasis.
- If state changes need motion, use existing `.cl-motion-state` only: opacity/color/background/border transitions, `--cl-dur-ui` 180ms or less.
- Panel/disclosure reveal may use existing `--cl-dur-panel` 260ms where already established.
- Do not animate reply sends, count ticks, auth failures, token reveal, raw details, or side-effect enqueue state.
- Reduced motion behavior must continue to collapse movement while preserving comprehension cross-fades.

## Verification Gates

Run these UI gates whenever `lib/cairnloop/web/**` or `priv/static/cairnloop.css` changes:

```bash
mix test test/cairnloop/web/token_drift_test.exs test/cairnloop/web/brand_token_gate_test.exs test/cairnloop/web/components_test.exs test/cairnloop/web/responsive_markup_test.exs test/cairnloop/web/motion_css_test.exs
```

Phase-specific verification additions:

- Component/CSS tests must prove new trust-state markup uses `.cl-*` components or tokenized BEM classes, not inline hex, raw `rgba()/hsl()`, Tailwind classes, or one-off styles.
- Copy/source scans must reject operator-visible raw Elixir terms, raw JSON, support message bodies, raw webhook payloads, secrets, placeholder Scrypath credentials, and `inspect(reason)` output.
- Responsive tests must cover Settings trust rows, Audit metadata disclosure, and Conversation evidence rail warnings at mobile and desktop widths when those surfaces change.
- Motion tests must prove no new layout-property animations and reduced-motion behavior for any new state transitions.
- Phase validation must still run the trust-boundary tests from `58-VALIDATION.md`, plus `mix compile --warnings-as-errors` and `mix ci.fast` before declaring the phase done.
- If docs/troubleshooting copy changes, run the relevant docs/source-scan test or add `test/cairnloop/docs_trust_test.exs` as planned in `58-VALIDATION.md`.

Manual review is limited to final prose inspection for calm, fail-closed, reason-forward copy. Browser/rendered behavior should be automated with targeted LiveView/component or Playwright checks when geometry, tap targets, or disclosure behavior matters.

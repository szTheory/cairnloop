<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser/customer identity -> Chat/Conversation | untrusted customer/session identity becomes durable customer_ref only |
| installer/example migrations -> adopter DB | generated and example schemas must create the identity field consistently |
| existing install -> additive upgrade path | host adds nullable customer_ref without repurposing host_user_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-58-01-01 | Elevation of Privilege | Chat.create_customer_conversation/1 | mitigate | customer token writes customer_ref only; host_user_id remains operator/governance identity per D-01 |
| T-58-01-02 | Repudiation | Conversation persistence | mitigate | additive customer_ref preserves separate evidence for customer/session identity without polluting operator attribution |
| T-58-01-03 | Tampering | installer/test-host/example migrations | mitigate | source-scan tests prove customer_ref exists in all conversation table creation paths |
| T-58-01-SC | Tampering | package installs | accept | no new npm/pip/cargo package installs in this plan |
</threat_model>

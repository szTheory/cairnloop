## Summary

Seven documentation files were reviewed: four new ExDoc guide files, a restructured README, a CHANGELOG entry, and mix.exs ExDoc wiring. The mix.exs ExDoc configuration is syntactically correct and the package `:files` list correctly includes `guides/`. The CHANGELOG entry is well-formed. The host integration guide's API references match the actual behaviour definitions exactly. The JTBD walkthrough stage descriptions match the code. Route names are consistent (`/support`, `/support/:id`) throughout.

Two blockers were found. The first is a compile-time crash: the quickstart router snippet passes `host_user_id:` as a `live_session` option, which Phoenix LiveView validates strictly and raises `ArgumentError` at compile time. The second is a missing migration step: the quickstart install flow never tells adopters to run the library's own migrations from `deps/cairnloop/priv/repo/migrations/`, meaning 15 library migrations (knowledge base, retrieval, governance, outbound, etc.) are silently skipped after a correct-looking `mix cairnloop.install && mix ecto.migrate`. Four additional warnings cover the docker-compose location claim, CHANGELOG link references, an ambiguous working-directory context in the Boot section, and an imprecise "via Oban" claim about the Notifier callbacks.

## Critical Issues

### CR-01: Quickstart Router Snippet Raises `ArgumentError` at Compile Time

**File:** `guides/01-quickstart.md:78`

**Issue:** The "Mount the Dashboard" snippet shows:

```elixir
cairnloop_dashboard("/", host_user_id: "demo_operator")
```

The `cairnloop_dashboard/2` macro passes its second argument directly to `live_session/3` as `opts`. Phoenix LiveView's `validate_live_session_opts/3` (confirmed in `phoenix_live_view/lib/phoenix_live_view/router.ex:307-312`) raises `ArgumentError: unknown live_session option ":host_user_id"` for any key other than `:session`, `:on_mount`, `:layout`, and `:root_layout`. This crashes router compilation. The example app avoids this by expanding the macro manually and using `session: %{"host_user_id" => "demo_operator"}`.

**Fix:**

```elixir
# guides/01-quickstart.md — Mount the Dashboard section
scope "/support", MyAppWeb do
  pipe_through :browser

  cairnloop_dashboard("/", session: %{"host_user_id" => "demo_operator"})
end
```

The macro comment in `lib/cairnloop/router.ex:7` already says "Host apps should provide `host_user_id` in the live session payload" — the snippet just needs to use the correct `session:` key.

### CR-02: Install Flow Silently Skips 15 Library Migrations

**File:** `guides/01-quickstart.md:19-49`

**Issue:** The quickstart install section describes `mix cairnloop.install` followed by `mix ecto.migrate` as sufficient to bring up the database. In practice, `mix cairnloop.install` (confirmed in `lib/mix/tasks/cairnloop/install.ex`) generates only a single `create_cairnloop_tables` migration for `cairnloop_conversations` + `cairnloop_messages`. The library ships 15 additional migrations in `priv/repo/migrations/` covering knowledge base, retrieval corpus, gap candidates, article suggestions, review tasks, tool proposals, tool approvals, MCP tokens, outbound bulk envelopes, and more.

`mix ecto.migrate` run without an explicit `--migrations-path` does not touch `deps/cairnloop/priv/repo/migrations/`. An adopter following the quickstart verbatim will boot the app and immediately hit `(Postgrex.Error) relation "cairnloop_knowledge_base_articles" does not exist` (or similar) the first time any non-chat feature is exercised.

The example app already handles this correctly: its `ecto.setup` alias runs:
```bash
mix ecto.migrate
mix ecto.migrate --migrations-path deps/cairnloop/priv/repo/migrations
```

**Fix:** Add a "Run library migrations" step in the quickstart after `mix deps.get`:

```bash
# Run host migrations (generated by the installer or written by hand)
mix ecto.migrate

# Run the Cairnloop library's own migrations
mix ecto.migrate --migrations-path deps/cairnloop/priv/repo/migrations
```

Also add this two-command pattern to the `ecto.setup` alias example shown in the troubleshooting guide's migration-order section, so adopters know to include it in their aliases from the start.

### WR-01: `docker-compose.yml` Claimed to Be in Example App but Lives at Repo Root

**File:** `guides/01-quickstart.md:12-15`

**Issue:** The quickstart says "the example app ships a `docker-compose.yml`" and immediately shows:

```bash
docker compose up -d db
```

`docker-compose.yml` lives at the repo root (`/docker-compose.yml`), not inside `examples/cairnloop_example/`. A reader who has `cd`'d into `examples/cairnloop_example/` (as both the quickstart intro and the JTBD walkthrough suggest) will get `no configuration file provided: not found` from Docker Compose.

**Fix:** Attribute the file to the repo root and add a working-directory note:

```markdown
If you need a containerized Postgres with pgvector, the repository ships a
`docker-compose.yml` at the repo root. From the repo root run:

\```bash
docker compose up -d db
\```
```

### WR-02: CHANGELOG.md Missing Keep-a-Changelog Link Reference Definitions

**File:** `CHANGELOG.md:8,20`

**Issue:** The file uses the Keep-a-Changelog section headers `## [Unreleased]` and `## [0.1.0] - 2026-05-25`, which are Markdown link references (`[Unreleased]` and `[0.1.0]` resolve to URLs in well-formed Keep-a-Changelog files). The Keep-a-Changelog specification expects link definitions at the bottom of the file:

```markdown
[Unreleased]: https://github.com/szTheory/cairnloop/compare/v0.1.0...HEAD
[0.1.0]: https://github.com/szTheory/cairnloop/releases/tag/v0.1.0
```

Without these, `[Unreleased]` and `[0.1.0]` render as literal bracketed text rather than links. The `package.links` entry already points to `https://hexdocs.pm/cairnloop/changelog.html`, so the rendered HexDocs page will show non-linked section headers.

**Fix:** Append the two link reference definitions to the end of `CHANGELOG.md`.

### WR-03: Quickstart Boot Section Omits Working Directory Context

**File:** `guides/01-quickstart.md:96-105`

**Issue:** The "Boot" section presents:

```bash
mix setup
mix phx.server
```

and tells the reader to visit `http://localhost:4000/support`. At no point in the quickstart has the reader been told to `cd examples/cairnloop_example`. A reader who followed the "Install" section (which is framed as setting up their own host app) will run `mix setup` in their own app directory, not in the example app directory — and will not see the seeded 12-16 conversations the rest of the guide describes.

The JTBD walkthrough correctly shows `cd examples/cairnloop_example && mix setup && mix phx.server` but the quickstart does not.

**Fix:** Add a working directory callout before the Boot commands:

```markdown
## Boot

The commands below are for the example app. Switch into it first:

\```bash
cd examples/cairnloop_example
mix setup
mix phx.server
\```
```

### WR-04: Host Integration Guide Overstates Oban Guarantee for `on_sla_breach`

**File:** `guides/03-host-integration.md:84`

**Issue:** The Notifier section states "Cairnloop calls each callback asynchronously via Oban, ensuring reliable retries and data consistency." This is accurate for `on_conversation_resolved/2` (called from `NotifyResolvedWorker`, a dedicated Oban worker) and `on_outbound_triggered/2` (called from `OutboundWorker`). However, `on_sla_breach/3` is called directly inside `Cairnloop.Workers.CheckSLA`'s `perform/1` function (confirmed in `lib/cairnloop/workers/check_sla.ex:25`). `CheckSLA` itself is Oban-backed, so the retry guarantee applies to the outer worker job, but the callback is a direct synchronous call inside that job — not dispatched to a separate dedicated worker. If `on_sla_breach/3` raises, it will fail the `CheckSLA` Oban job and trigger a retry of the entire SLA check, which is functionally correct but different from the dedicated-worker pattern the other two callbacks use.

**Fix:** Tighten the claim to reflect actual delivery semantics:

```markdown
`Notifier` is the business-logic integration point. Cairnloop calls these callbacks from
within Oban workers — `on_conversation_resolved/2` and `on_outbound_triggered/2` each have
a dedicated Oban worker, while `on_sla_breach/3` is called directly within the SLA
countdown worker. Raising in any callback retries the enclosing Oban job.
```

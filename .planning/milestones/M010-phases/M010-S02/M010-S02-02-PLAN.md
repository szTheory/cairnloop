## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| request path -> background worker | Suggestion generation must not run inline or duplicate for the same grounded input |
| retrieval evidence -> generation prompt | Only explicitly grounded evidence with canonical anchors may justify generated KB content |
| stale-pressure projection -> revision entrypoint | Weak or noisy article-linked failures could otherwise over-promote revisions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-M010-S02-04 | D | `GenerateArticleSuggestion` queueing | mitigate | Use Oban uniqueness keyed by entrypoint identity plus `evidence_digest` so repeated clicks or retries do not spawn duplicate generation for the same grounded input. |
| T-M010-S02-05 | T | stale-article revision gate | mitigate | Require at least two article-linked failures within 30 days plus a published `base_revision_id`, and derive links from canonical citation anchors rather than timestamps or title similarity. |
| T-M010-S02-06 | S | generation grounding boundary | mitigate | Block success when canonical citation anchors are missing or grounding is weak, persist a durable `:failed` suggestion with explicit reason, and keep assistive-only evidence from grounding normal KB content. |
</threat_model>

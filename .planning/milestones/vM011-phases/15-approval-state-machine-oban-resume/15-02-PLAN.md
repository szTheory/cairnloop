<objective>
Build the two Oban workers (M011-S03-03) that make approval resume and expiry durable, async, and
fail-closed. The `ApprovalResumeWorker` is THE Phase-15 deliverable: it re-calls the pure
`Governance.validate/3` (re-validation is free by construction, P13 D-15) + `Policy.resolve/3`
against CURRENT context, and on pass transitions to the `:execution_pending` seam — it STOPS there
and NEVER calls `run/3` (that is the Phase 16 hook). On failure it invalidates fail-closed. A lazy
`expires_at` guard is the belt-and-suspenders that makes a missed scheduled sweep harmless. The
`ApprovalExpiryWorker` is the scheduled `:pending → :expired` flip. Both clone `SlaCountdownWorker`
exactly. This plan also extends the `Policy.resolve/3` PDP seam in place (the actor-scope factor).

Purpose: The re-validate-before-execute gate is the rubric's "double-check policy proof" — an
approval valid yesterday is re-checked against today's scope/policy at resume (Terraform "stale
plan" semantics). Workers are self-contained new files over the Wave-1 schema; the facade's
`approve/3` (plan 15-03) references `ApprovalResumeWorker`, so the worker must exist first.
Output: 2 Oban workers + extended Policy.resolve/3; Wave 0 worker tests turn green via MockRepo.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| approved approval → execution (deferred to Phase 16) | An approval valid at decision time could be stale at resume if policy/scope changed; must re-validate. |
| Oban job args (JSONB) → worker rehydration | String-keyed snapshot keys rehydrated to atoms could exhaust the atom table if attacker-influenced. |
| host Oban runtime → library job enqueue | The host owns Oban; the library may run where Oban is unconfigured. |

<objective>
Implement MCP Token management in the Operator Settings Surface.

Purpose: Provide a secure interface for operators to provision, edit, and revoke MCP tokens for external tools, ensuring the raw token is only ever displayed once.
Output: Updates to `SettingsLive` with token listing, creation form, single display of raw token, editing capability, and revocation functionality. Updates to `Cairnloop.MCP` for token modification.
</objective>

<task type="auto">
  <name>Task 1: Active MCP Tokens List and Empty State</name>
  <files>lib/cairnloop/web/settings_live.ex</files>
  <action>
    Update `SettingsLive` to query and display active tokens.
    1. Add `alias Cairnloop.MCP.Token` and `import Ecto.Query`.
    2. In `mount/3`, add a repo lookup: `repo = Application.fetch_env!(:cairnloop, :repo)`.
    3. Query active tokens: `tokens = repo.all(from t in Token, where: is_nil(t.revoked_at), order_by: [desc: t.inserted_at])`. Assign to socket (`@tokens`), and assign `@new_raw_token` as `nil`.
    4. In `render/1`, add a `.cl-card` section for "MCP Authentication".
    5. If `@tokens` is empty, render the empty state: heading "No MCP tokens active" and body "Add a token to enable external tool capabilities for this app." (per UI-SPEC).
    6. If tokens exist, render a list displaying the `name` and a masked visual representation (e.g., `cl_mcp_***`).
  </action>
  <verify>
    <automated>mix test test/cairnloop/web/settings_live_test.exs</automated>
  </verify>
  <done>Active tokens are queried directly via Ecto and displayed, with empty state correctly adhering to UI-SPEC.</done>
</task>

<task type="auto">
  <name>Task 2: Issue MCP Token and Single Display</name>
  <files>lib/cairnloop/web/settings_live.ex</files>
  <action>
    Add token creation capability ensuring the raw token is displayed exactly once.
    1. Add a form to the "MCP Authentication" section with a text input for `name` and a submit button with the copy "Add MCP Token" (primary CTA).
    2. Implement `handle_event("create_token", %{"name" => name}, socket)`:
       - Call `Cairnloop.MCP.issue_token(%{name: name})`.
       - On `{:ok, token, raw_token}`, prepend the new token to the `@tokens` list, assign `new_raw_token: raw_token`, and put an `:info` flash message.
       - On `{:error, _}`, put an `:error` flash message.
    3. In `render/1`, conditionally display `@new_raw_token` inside an alert/notice block if it is not `nil`. Emphasize that it must be copied now because it will not be shown again (addresses Pitfall 1 from RESEARCH.md).
  </action>
  <verify>
    <automated>mix test test/cairnloop/web/settings_live_test.exs</automated>
  </verify>
  <done>Creating a token delegates to `Cairnloop.MCP` and the raw string is temporarily available in the UI but not persisted to state on refresh.</done>
</task>

<task type="auto">
  <name>Task 3: Edit MCP Token</name>
  <files>lib/cairnloop/mcp.ex, lib/cairnloop/web/settings_live.ex</files>
  <action>
    Add the ability for an operator to edit an existing token's name.
    1. In `Cairnloop.MCP`, add `update_token/2` to allow updating token fields (specifically `:name`), applying a changeset and saving to the repo.
    2. In `SettingsLive`, add an inline edit form or modal trigger to the token list UI allowing the operator to rename the token.
    3. Implement `handle_event("update_token", %{"id" => id, "name" => new_name}, socket)`:
       - Find the token in `@tokens`.
       - Call `Cairnloop.MCP.update_token(token, %{name: new_name})`.
       - On success, update the token in the `@tokens` list and display a success flash.
  </action>
  <verify>
    <automated>mix test test/cairnloop/mcp_test.exs test/cairnloop/web/settings_live_test.exs</automated>
  </verify>
  <done>Operator can edit the name of an existing MCP token, persisting changes via `Cairnloop.MCP.update_token/2`.</done>
</task>

<task type="auto">
  <name>Task 4: Revoke MCP Token</name>
  <files>lib/cairnloop/web/settings_live.ex</files>
  <action>
    Allow operators to revoke tokens with destructive confirmation.
    1. In the token list UI, add a "Revoke" button to each item.
    2. Add `phx-click="revoke_token"`, `phx-value-id={token.id}`, and the data attribute `data-confirm="Revoke Token: Are you sure? Active integrations using this token will fail immediately."` (per UI-SPEC).
    3. Implement `handle_event("revoke_token", %{"id" => id}, socket)`:
       - Find the token in the `@tokens` list.
       - Call `Cairnloop.MCP.revoke_token(token)`.
       - On success, remove it from the `@tokens` list in assigns, assign `@new_raw_token` to `nil` (clearing any visible raw token), and display a success flash.
  </action>
  <verify>
    <automated>mix test test/cairnloop/web/settings_live_test.exs</automated>
  </verify>
  <done>Revoke events call `Cairnloop.MCP.revoke_token/1`, require confirmation, and visually remove the token from the list upon success.</done>

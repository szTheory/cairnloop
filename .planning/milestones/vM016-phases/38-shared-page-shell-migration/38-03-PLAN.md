<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `return_to` path → operator-visible crumb label | The `return_to` string is the only externally-influenced input. It is ALREADY verified upstream (decoded from a signed `handoff` token at `editor.ex:147,207-214`) before it ever reaches this presenter. This plan does NOT decode or trust raw params — it receives the already-verified value and renders only a DERIVED label from its path shape. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-05 | Information disclosure / Spoofing | origin crumb label derivation | mitigate | The presenter NEVER returns the raw `return_to` path as a label — it returns only the humanized noun "Conversation"/"Suggestions" (copy rule + ASVS V5 output encoding). Enforced by a negative-copy unit test asserting no label equals the raw path. |
| T-38-06 | Tampering (open-redirect via back link) | `href: return_to` in the items list | accept | The `href` reuses the already-verified `return_to` (signed-token-derived; the existing `editor.ex:287` already renders it as a `navigate=` href). No NEW token plumbing and no unverified param is introduced — this presenter only formats a value verified upstream. |
| T-38-SC | Tampering | npm/pip/cargo installs | accept | No packages installed — pure in-repo Elixir module + ExUnit test. No install task in this plan. |

No `high`-severity threats: the one security-relevant surface (path → label) is mitigated by rendering a derived label only, asserted by a negative-copy test; the back-link href reuses an upstream-verified value.

<objective>
Build the backend read seam that lets an Audit Log row link to its subject conversation
(THREAD-02) and lets the audit log be filtered to one governed action's trail (THREAD-03a
backing read). This is the domain layer only — no LiveView/markup changes here; Plan 03 wires
the web layer on top of these reads.

Purpose: AuditLogLive consumes the pluggable `Cairnloop.Auditor` behaviour, whose default impl
flattens each event to a plain map that drops the proposal/ids. Without enriching that map,
no audit row can resolve its subject (RESEARCH correction #1 / Pitfall 1).

Output: enriched auditor normalization map (conversation_id + proposal_id per event), an
optional `proposal_id:` filter on `Governance.list_action_events/1`, and a total presenter
helper that returns a scope-relative subject href (or nil for fail-closed absence).
</objective>

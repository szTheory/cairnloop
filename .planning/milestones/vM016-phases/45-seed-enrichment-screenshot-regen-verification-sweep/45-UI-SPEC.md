| Property | Value |
|----------|-------|

## Phase UI Boundary

Phase 45 is an evidence phase. It does not redesign the cockpit; it makes existing states visible, captures them in both themes, and records visual acceptance.

| Surface | Contract |
|---------|----------|
| Seeded operator states | Enriched demo data must reveal happy, empty, error, dense, and boundary states through existing UI routes and components |
| Screenshot pipeline | Regenerate explicit light and explicit dark captures for operator/admin screens only |
| Visual acceptance ledger | Add `45-VISUAL-ACCEPTANCE.md` with one row per captured screen and theme |
| Verification artifacts | Use screenshots as milestone evidence; use automated tests and gates for behavior |

Operator/admin screenshot matrix:

| Screen/state | Required evidence |
|--------------|-------------------|
| Cockpit home | Final-brand home with queue-first hero, secondary trail band, and health chip |
| Inbox/list triage | Open/resolved/bulk-ready rows with readable status and selection affordances |
| Conversation workspace | Timeline, evidence rail, density/disclosure controls, and reply area |
| AI draft approval | Draft decision state with clear approve/send action and non-raw explanation |
| Pending governed action | Approval-required governed action with risk, confidence, policy outcome, and reason |
| Executed governed action | Completed action with event history and user-facing result copy |
| Rejected governed action | Rejected state with human-readable reason and visible timestamp |
| Deferred governed action | Deferred state with reason and follow-up context |
| Resolved/outbound recovery | Recovery surface with calm reason-forward copy |
| Bulk recovery modal/refusal | Capture only if seeded; modal/refusal must fit viewport and not obscure row context incoherently |
| Knowledge index | Published and draft article state visible without backend terms |
| Knowledge gaps | Gap candidates with signal counts and source context |
| Knowledge review/suggestions | Rejected, deferred, approved-ready, and published ReviewTask states |
| Article editor | Draft article with unpublished draft revision visible |
| Audit log | Varied audit timestamps across roughly 14 days and governed-action events |
| Settings | Masked MCP token rows such as `cl_mcp_***`; no raw token value visible |

Exclude the demo index, marketing pages, and customer chat widget from Phase 45 visual acceptance. If existing docs still need those screenshots, they may remain as docs captures, but they must not count toward the Phase 45 acceptance ledger.

## Component Contract

Use existing shared components and CSS classes. New markup must compose them rather than re-authoring local variants.

| Need | Required primitive |
|------|--------------------|
| Page framing | `cl_shell`, `cl_page`, `cl_breadcrumb` |
| Primary cockpit count | `cl_hero` |
| Secondary counts | `cl_stat` |
| Actions | `cl_button` variants `primary`, `ghost`, `danger`, `default` |
| Status labels | `cl_chip` or `cl_status_cell` with icon + text |
| Persistent status | `cl_banner` |
| Flash notices | `cl_flash` |
| Empty states | `cl_empty` |
| Cards and evidence panels | `cl_card`, `cl_source_card`, `cl_fact_list` |
| Progressive disclosure | `cl_disclosure` with native `<details>`/`<summary>` and stable `id` |
| Toggles | `cl_switch` for binary settings; preserve visible label |
| Tables | `.cl-table` inside `.cl-table-scroll` with accessible region semantics |

Do not introduce hardcoded color literals in render `.ex` files, inline `style` colors, raw `rgba()`/`hsl()`, hidden color-only state, new icon packages, live-text logo recreation, custom button classes, or Tailwind/daisyUI restyling for Cairnloop operator surfaces.

## Spacing Scale

Declared values use the approved Cairnloop 4px rhythm. New or touched UI/evidence surfaces should use these values through existing `--cl-space-*` tokens and component classes.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, focus offset, tiny inline spacing |
| sm | 8px | Compact metadata rows, chip gaps, field stacks |
| md | 16px | Default panel padding, page gutter, ledger table spacing |
| lg | 24px | Section padding, modal interior spacing, major row groups |
| xl | 32px | Page/content gaps and two-column cockpit spacing |
| 2xl | 48px | Empty-state padding and large visual breaks |
| 3xl | 64px | Page-level vertical spacing only |

Exceptions: existing dense admin tokens `12px`, `20px`, and `40px` may remain where already baked into `priv/static/cairnloop.css`; do not introduce them as new Phase 45 spacing choices outside existing component classes. Interactive touch targets remain at least `44px` where visible. Existing fixed layout values remain authoritative: `--cl-content-max: 1200px`, `--cl-rail-width: 352px`, screenshot viewport `1440x900`, and device scale `2`.

Typeface usage:

| Typeface | Usage |
|----------|-------|
| Atkinson Hyperlegible Next / Atkinson Hyperlegible / system sans fallback | Operator UI labels, body copy, controls, status text |

Existing component-defined display/count exceptions may remain: `cl_hero__count` at `48px` and `cl_stat__count` at `32px`. Do not add new font families, CDN fonts, negative letter spacing, viewport-scaled type, or live-text logo recreation.

## Color

Use the final vM017 Refined palette from `priv/static/cairnloop.css`. Screenshot acceptance must verify both light and dark theme values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F4EEE2` light canvas, `#101614` dark canvas | Page backgrounds and broad shell surfaces |
| Secondary (30%) | `#FAF5EB` light surface, `#FFFFFF` light raised surface, `#141B19` dark surface, `#1F2C28` dark raised surface | Cards, nav, panels, inputs, modals, tables |
| Accent (10%) | `#A8492A` light copper, `#D98A4A` dark copper | Primary action, hero count, focus ring, active route marker, selected/current edge, C3.6 logo ring |
| Destructive | `#B54C36` light danger, `#C96A55` dark danger | Reject/error/failed states and destructive-action styling only |

Accent reserved for: the C3.6 crowning-loop ring in committed logo assets, primary buttons, the `cl_hero` primary count, active route markers, focus rings, selected/current borders, and small route-marker annotations. Do not use copper for every link, every heading, decorative fills, generic table chrome, or broad backgrounds.

Semantic colors:

| Token | Usage |
|-------|-------|
| `--cl-success` and success triplet | Healthy, published, completed, all-caught-up states |
| `--cl-info` and info triplet | Links, retrieval/source context, informational banners |
| `--cl-warning` and warning triplet | Deferred, pending, risk-needs-review, budget/scope warnings |
| `--cl-danger` and danger triplet | Rejected, failed, policy-blocked, refusal, error states |
| `--cl-ai` and AI triplet | AI/evaluation/source-of-draft states only |

Every meaningful status must pair color with visible text and a distinct icon or shape. Light and dark screenshots must show intentional theme contrast, not stale light-theme surfaces under dark filenames. No off-brand gradients, old palettes, arbitrary recoloring, stale logo assets, or low-contrast logo backgrounds are allowed.

## Screenshot And Visual Evidence Contract

Screenshot output for Phase 45 must be explicit by theme.

| Requirement | Contract |
|-------------|----------|
| Theme selection | Capture both `light` and `dark` for every required operator/admin shot |
| Browser preference | Set Playwright `colorScheme` to the active theme |
| App theme state | Set `document.documentElement.dataset.theme` and `localStorage["phx:theme"]` to the active theme before route hydration |
| Motion | Use reduced motion and stabilization CSS for static screenshots; live motion remains covered by E2E |
| Readiness | Wait for LiveView connection and a screen-specific visible selector; do not rely only on sleeps |
| File layout | Use `guides/assets/light/NN-name.png` and `guides/assets/dark/NN-name.png` for Phase 45 acceptance outputs |
| Capture scope | Use full-page capture for pages and viewport capture for modals/overlays |
| Pixel gating | No CI pixel/snapshot gate, Percy, Chromatic, or Storybook visual review |

The visual acceptance ledger must contain one row per screenshot and theme with: screen/state, theme, screenshot path, final-brand token/logo check, visible state check, hierarchy/density check, accessibility/non-color-only check, copy check, pass/fail, and notes.

Acceptance checks:

| Dimension | Pass condition |
|-----------|----------------|
| Brand fidelity | Final vM017 tokens and committed logo assets are respected; no stale logo or old palette appears |
| Theme correctness | Light and dark captures both look intentional and use the right semantic surfaces |
| State visibility | The seeded state named by the matrix is visible without manual setup |
| Hierarchy | Primary work, secondary context, and raw detail are visually distinct |
| Density | Operator cockpit remains scannable; dense data does not overlap or clip |
| Accessibility | Status is never color alone; focus/tap affordances remain conventional where visible |
| Copy | User-facing copy is calm, reason-forward, and free of raw Elixir atoms, raw JSON, table names, or backend implementation terms |

## Interaction Contract

Phase 45 must preserve existing operator workflows while making more states visible.

| Interaction | Contract |
|-------------|----------|
| Theme toggle | Existing `phx:theme` / `data-theme` behavior remains unchanged; screenshot automation forces theme state without changing product behavior |
| Rail disclosure | Native `<details>` disclosure remains browser-owned and must not snap closed on LiveView patches |
| Rail density | Existing `localStorage["cl:rail:density"]` behavior remains browser-owned |
| Governed actions | Approval/reject/defer states must be visible with risk, confidence, policy outcome, timestamp, and human reason; raw tool params stay behind explicit expanders |
| Knowledge review | Visible queue state comes from `ReviewTask`, not invented `ArticleSuggestion` statuses |
| MCP tokens | Settings shows masked token handles only; raw token material must never appear in UI, logs, screenshots, tests, or ledger notes |
| Bulk modal/refusal | Modal content remains keyboard-reachable, visible within the `1440x900` viewport, and reason-forward |
| Tables | `.cl-table-scroll` remains accessible on constrained widths and visible screenshots must not hide important status behind horizontal clipping |
| Motion | No new motion classes for screenshots. Existing transform/opacity-only and `prefers-reduced-motion` contracts remain in force |

Browser-visible behavior that changes during implementation must be covered by `PhoenixTest.Playwright.Case` E2E, not human UAT. Screenshots are reviewed evidence and docs assets; they are not a substitute for behavior gates.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Regenerate screenshots |
| Empty state heading | No evidence captured |
| Empty state body | Run the seeded demo, regenerate light and dark operator screenshots, and record each screen in the visual acceptance ledger. |
| Error state | Screenshot capture failed. Check the reported route, selector, and seeded state, then rerun the capture. |
| Destructive confirmation | none; Phase 45 must not introduce new destructive product actions |

Required seed/evidence copy patterns:

| State | Copy contract |
|-------|---------------|
| Rejected governed action reason | `Rejected: available evidence does not justify this action.` |
| Deferred governed action reason | `Deferred: waiting for a policy owner to confirm scope.` |
| Higher-risk demo tool label | `High-risk demo action` with visible `Approval required` status |
| KB rejected review note | `Rejected: source evidence was not strong enough to publish.` |
| KB deferred review note | `Deferred: needs policy-owner review before publishing.` |
| Draft article state | `Draft saved` and `Unpublished changes` where applicable |
| MCP token names | `Demo docs integration` and `Demo audit export`; visible values must be masked as `cl_mcp_***` |
| Ledger pass label | `PASS` |
| Ledger fail label | `FAIL` |

Voice: calm, factual, operator-grade, and reason-forward. Avoid launch copy, generic AI claims, cute/mascot language, raw atoms such as `:high_write`, raw JSON, schema/table names, or implementation exception text. State labels must be humanized before display.

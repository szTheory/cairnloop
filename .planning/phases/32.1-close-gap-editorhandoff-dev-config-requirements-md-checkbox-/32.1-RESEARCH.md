### Pattern 1: Fail-Closed Secrets
**What:** Using `System.fetch_env!/1` in production to ensure the application fails to start if a required secret is missing.
**When to use:** Whenever injecting cryptographic keys or essential secrets in a release environment.
**Example:**
```elixir
config :cairnloop, Cairnloop.KnowledgeAutomation.EditorHandoff,
  secret_key_base: System.fetch_env!("CAIRNLOOP_HANDOFF_SECRET_KEY_BASE")
```

### Anti-Patterns to Avoid
- **Silent Failures:** Using `System.get_env/1` for required secrets which might evaluate to `nil` and cause delayed crashes at runtime. Always use `fetch_env!/1` for required configuration.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Env Config | Custom `read_env()` function | `System.fetch_env!/1` | Built-in functionality natively handles missing variable exceptions reliably. |
| Path Traversal | Custom recursive traversal | `Path.wildcard("**/*.ex")` | Native `Path.wildcard` supports deep recursive lookups efficiently. |

## Common Pitfalls

### Pitfall 1: Non-recursive Wildcards
**What goes wrong:** Tests designed to gate all files in a directory only process the top level.
**Why it happens:** Using `*.ex` instead of `**/*.ex`.
**How to avoid:** Always use `**/*.ex` when structural assertions require deep evaluation.

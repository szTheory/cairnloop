<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser `?status=` query param → InboxLive | Untrusted, attacker-controllable URL input crosses into the LiveView. |
| InboxLive selection state → bulk actions | Bulk actions must operate only on currently-visible rows. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-03 | Denial of Service | `normalize_status/1` | mitigate | Explicit string-literal whitelist (`"resolved"` only) — NEVER `String.to_existing_atom`/`String.to_atom` on `params["status"]`. Unknown input → `nil` → unfiltered. Prevents atom-table exhaustion and ArgumentError crashes from crafted URLs. Proven by the garbage-input whitelist test (Task 1). |
| T-39-04 | Tampering / Injection | `?status=` → Ecto `where` | mitigate | Param is normalized to a known atom (or nil) before reaching `Chat.list_conversations(status:)`; the facade binds it as a parameterized `^status` pin (Plan 01). No raw string ever interpolated into SQL. |
| T-39-05 | Repudiation / Integrity | `prune_selected_ids/2` | mitigate | Routed through BOTH `handle_params/3` and the PubSub handler (D-04) so a bulk action can never operate on rows that left the filtered view — prevents silent-count-inflation / acting on a stale selection. |
| T-39-SC | Tampering | hex/npm/cargo installs | accept | No packages installed (RESEARCH: not applicable). |
</threat_model>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| customer browser ↔ WidgetSocket | The new `/widget` socket mount is the WebSocket transport boundary. Token validation happens in `WidgetSocket.connect/3` (demo mode — any binary token). |
| JS hook ↔ LiveView (pushEvent / push_event) | Channel between the colocated hook and the LV process. The hook never trusts inbound payloads from the LV beyond the `widget:send` forwarding; the LV never trusts the hook's `channel_status` value beyond the four pre-existing atoms. |
| ChatLive `handle_event` payload → server state | Customer-supplied `message` text is the only user-tainted field; `id` (conversation_id from hook) is server-issued via the join reply. |
| ChatLive `handle_info` ↔ Cairnloop.PubSub | Topic-namespaced per conversation; the LV only subscribes to its own conversation's topic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-03-01 | Tampering | Customer crafts `phx-submit="send_message"` with oversized `message` text | accept | Inherits Phase 28 scope: `Cairnloop.Message.changeset/2` validates presence but not max length. Bounded-length validation is a future follow-up. |
| T-28-03-02 | Tampering / XSS | Operator's reply content rendered in customer's `/chat` | mitigate | HEEx `<%= msg.content %>` auto-escapes — V5 ASVS. The bubble template uses interpolation, not `raw/1`. Acceptance criteria: no `raw(` calls in the rewritten chat_live.ex (`grep -c "raw(" examples/cairnloop_example/lib/cairnloop_example_web/live/chat_live.ex` returns `0`). |
| T-28-03-03 | Information Disclosure | Customer subscribes to another conversation's PubSub topic | mitigate | ChatLive subscribes ONLY to `"conversation:#{id}"` where `id` was issued by the server's `WidgetChannel.join/3` (Plan 02). The customer browser cannot subscribe to an arbitrary conversation — the LV does not accept conversation_id from the URL or any handle_event payload that the customer can forge. |
| T-28-03-04 | Denial of Service | JS hook sends `channel_status` with an arbitrary atom name (atom-table flooding) | mitigate | `handle_event("channel_status", %{"status" => status}, socket)` uses `String.to_existing_atom(status)` — pre-existing atoms only (`:connecting`, `:connected`, `:disconnected`). Unknown strings raise `ArgumentError` and the LV process crashes cleanly (which is acceptable — it is the customer's own browser tab, not a shared resource); the LV supervisor restarts it. | T-28-03-05 | Denial of Service | Customer rapid-fires `phx-submit="send_message"` to flood the channel + worker queue | accept | Phase 28 inherits the existing channel surface; Phoenix Channels enforce per-channel message rate limits at the transport layer (default `cowboy` limits). | T-28-03-06 | Spoofing | Colocated hook's `data-token` attribute is visible in DOM and tamperable | accept | Demo mode — `WidgetSocket.connect/3` accepts any binary token per the documented "T-M001-01" mitigation comment. The token serves only as a per-session identifier in the demo; production adopters bring their own signed-token verification. CLAUDE.md "host-owned posture" — out of Phase 28 scope. |
| T-28-03-07 | Repudiation | Customer-side optimistic message is never persisted if the server-side `ProcessMessage` worker fails | accept | The optimistic append shows immediately, but if the worker fails the message is lost from the operator's view. The customer would see the orphan in their thread. This is acknowledged behavior — full reliability requires explicit error UI ("Your message could not be sent. ...") wired to a hook-side `socket.onError` / `channel.push().receive("error", ...)` callback, which the UI-SPEC §1d copy contract specifies. The plan instructs the hook to push `channel_status: disconnected` on error; full per-message error reporting is acceptable to defer to a future phase. |

Three surfaces need UI work in Phase 28:

1. **`CairnloopExampleWeb.ChatLive`** (`/chat`) — customer-facing chat widget, currently 51 LOC mock. Rewrite to push messages through `WidgetChannel` and receive operator replies via PubSub. Remove `Process.send_after(self(), :bot_reply, 1000)`.

## Component Inventory

### 1. Chat Panel (`/chat` — `CairnloopExampleWeb.ChatLive`)

**Structure:** max-w-2xl centered container, `.cl-card` panel (Warm Stone background, 1px Granite border, 14px radius per `--cl-radius-lg`), full-viewport-height-minus-header flex column.

**Sub-components:**

#### 1a. Panel Header
- Text: "Cairnloop Support" at 18px / weight 600 / `var(--cl-text)`.
- Connection state indicator: small dot (8px × 8px) + label. States:
  - Connecting: `var(--cl-text-muted)` dot, label "Connecting…"
  - Connected: `var(--cl-success)` dot, label "Connected" (dot + text — never color alone; brand §7.5)
  - Disconnected: `var(--cl-danger)` dot, label "Disconnected — reconnecting"
- Separator: 1px `var(--cl-border)` below header.

#### 1b. Message Thread Area
- Scrollable flex-1 region with `overflow-y-auto`.
- Empty state: centered italic text "How can we help you today?" at 15px, `var(--cl-text-muted)`, `mt-10`.
- Customer message bubble: right-aligned, `var(--cl-primary)` background, `var(--cl-primary-text)` (`#FFFFFF`) text, `--cl-radius-lg` corners, `px-4 py-2`, max-w-[75%].
- Operator reply bubble: left-aligned, `var(--cl-surface-raised)` background, `var(--cl-text)` text, `1px var(--cl-border)` border, `--cl-radius-lg` corners, `px-4 py-2`, max-w-[75%].
- Timestamp under each bubble: 13px, `var(--cl-text-muted)`, no clock-emoji.
- "Waiting on operator" status row: appears after last customer message when no operator reply yet. Text: "Message sent — waiting on operator." 13px, `var(--cl-text-muted)`, left-aligned, NOT styled as a bubble.
- Pending delivery indicator: while `phx-submit-loading` is active on the form, show a subtle opacity transition (0.5) on the Send button text. Do not show a typing animation (brand §15.3 "no typing indicator unless real streamed response").

#### 1c. Message Input Form
- `phx-submit="send_message"` form.
- `<input type="text" name="message">`: flex-1, `var(--cl-surface-raised)` background, `1px var(--cl-border)` border, `--cl-radius-md` (10px) corners, `px-3 py-2`, placeholder "Type your message…", focus ring: `2px var(--cl-focus)` with 2px offset.
- Send button: `var(--cl-primary)` background, `var(--cl-primary-text)` text, `--cl-radius-md` corners, `min-h-[44px] px-4 py-2`, label "Send", `hover:opacity-90`, disabled when input is empty or form is submitting.
- Separator: 1px `var(--cl-border)` above form area.

#### 1d. Connection / Channel State (LiveView assigns)
- `@channel_status` :: `:connecting | :connected | :disconnected` — drives header indicator.
- `@messages` :: list of `%{role: :customer | :operator, content: String.t(), inserted_at: DateTime.t()}` — drives thread.
- `@pending` :: boolean — true while a customer message has been pushed but no operator reply received.

#### 1e. JavaScript Hook (client-side)
- A `WidgetChat` LiveView JS hook must open the `WidgetSocket` connection and join `"widget:lobby"` using `phoenix.js` channel primitives.
- Token flow: pass `token: "demo_customer"` in connect params for the example app (see `WidgetSocket.connect/3` — any binary token is accepted in demo mode).
- On channel `"new_operator_reply"` event received from server: push a `phx-hook` event to the LiveView process so the LV assigns update.
- Keep the hook minimal — no framework, no npm (existing pattern in this project).

## Interaction Contract

### Message Send Flow

1. Customer types in input and submits form (`phx-submit="send_message"`).
2. LiveView `handle_event/3` pushes the message over the JS channel (`"new_message"` event) via the `WidgetChat` hook.
3. Message is immediately appended to `@messages` as optimistic UI with `:customer` role.
4. `@pending` flips to `true`. "Waiting on operator." row appears below last customer message.
5. Server side: `WidgetChannel.handle_in("new_message", ...)` enqueues `ProcessMessage` worker, replies `:ok`.
6. `ProcessMessage` worker creates a `Cairnloop.Chat` message + broadcasts on PubSub.
7. `ConversationLive` on the operator side reloads and shows the new message.
8. Operator types a reply in `ConversationLive` and submits.
9. `Chat.reply_to_conversation/2` creates a message + broadcasts on `"conversation:#{id}"`.
10. Customer's `ChatLive` receives the broadcast (via PubSub subscription set up on `connected?(socket)`) and appends operator message with `:operator` role.
11. `@pending` flips to `false`. "Waiting on operator." row disappears.

### Channel Connect Flow

1. On LiveView connect, `WidgetChat` JS hook opens `WidgetSocket` at the endpoint's canonical path.
2. Hook joins `"widget:lobby"` (unauthenticated) or `"widget:{conversation_id}"` (if conversation already created).
3. `@channel_status` transitions: `:connecting` → `:connected` on `onOpen`; `:disconnected` on `onError/onClose`.
4. On disconnect, Phoenix channel client auto-reconnects; UI shows "Disconnected — reconnecting" until `onOpen` fires again.

### Accessibility

- Focus ring: `2px solid var(--cl-focus)` with `2px` offset on input and Send button.
- Send button disabled state uses both `disabled` attribute and `opacity-50` visual (never color alone — brand §7.5).
- Connection state indicator pairs dot color with text label — never color alone.
- `role="log"` on message thread `<div>` with `aria-live="polite"` so screen readers announce new messages.
- `aria-label="Chat message"` on each message bubble.
- `aria-label="Type your message"` on the input.

## Motion Contract

- Message append: no animation — instant DOM insert. (Brand §15.1: keep under 180ms; for appended DOM nodes in LiveView, avoid explicit transition to prevent jank on rapid messages.)
- Form submit state: `phx-submit-loading` class on Send button reduces opacity to 0.5 via `phx-submit-loading:opacity-50` Tailwind variant (already defined in `app.css` line 115).
- Connection state dot: no pulsing animation (brand §15.3 "no infinite pulsing around AI cards" — applies by extension to connection indicators).
- Panel reveal: none — page loads with panel already visible.

Source: Brand book §15.1, §15.3.

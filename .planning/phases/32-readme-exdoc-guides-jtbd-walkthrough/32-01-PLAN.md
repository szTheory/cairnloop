Install task (lib/mix/tasks/cairnloop/install.ex) — what `mix cairnloop.install` does:
- Adds dep: `{:cairnloop, "~> 0.1.0"}` via `Igniter.Project.Deps.add_dep`.
- Calls `Igniter.Libs.Ecto.select_repo()`; if it returns nil, emits the issue text:
  "No Ecto repo found. Please create a migration manually for cairnloop tables."
- Otherwise generates a `create_cairnloop_tables` migration (`on_exists: :skip`) creating
  `cairnloop_conversations` and `cairnloop_messages` tables against the detected repo.

Dashboard mount (lib/cairnloop/router.ex — `cairnloop_dashboard/2` macro):
- Mounts inbox at `<path>/`, conversation at `<path>/:id`, plus KB routes
  (`/knowledge-base`, `/knowledge-base/gaps`, `/knowledge-base/suggestions`,
  `/knowledge-base/:id/edit`) and `/settings`.
- Example app mounts with `cairnloop_dashboard("/", host_user_id: "demo_operator")` inside
  `scope "/support"` → inbox is `/support`, a conversation is `/support/:id`.

ROUTE CONVENTION (Pitfall 1 — load-bearing): adopter guides MUST use `/support` and
`/support/:id`. NEVER document `/inbox` or `/governance/:id` (those are integration-test-host
routes, not shipped routes).

Version pin (Pitfall 5): always `{:cairnloop, "~> 0.1.0"}` — matches install.ex and mix.exs
`version: "0.1.0"`. Do NOT guess a different version.

mix.exs `test.setup` alias (migration ordering — for troubleshooting):
- `ecto.migrate --migrations-path priv/test_host/migrations --migrations-path priv/repo/migrations`
- Host-owned tables (version-prefixed `20260101…`) must precede library migrations so the
  library's FKs to `cairnloop_conversations` resolve. Document this ordering requirement.

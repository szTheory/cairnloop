## Phase Details

### Phase 18: Release Gate & Hex.pm Publish
**Goal**: Cairnloop is publicly discoverable, installable, and documented on hex.pm — CI is clean and the codebase is tagged at v0.1.0
**Depends on**: Nothing (first phase of vM012; hygiene gate in STATE.md must be complete)
**Requirements**: REL-01, REL-02, REL-03, REL-04, REL-05, REL-06
**Success Criteria** (what must be TRUE):
  1. An adopter can add `{:cairnloop, "~> 0.1"}` to `mix.exs` and `mix deps.get` resolves successfully from hex.pm
  2. `hexdocs.pm/cairnloop` serves the API documentation generated by ExDoc
  3. The package page at `hex.pm/packages/cairnloop` shows correct metadata (description, license, links, maintainers)
  4. CHANGELOG.md is committed and covers vM009–vM012 with dates and feature summaries
  5. CI passes green on origin/main for both integration and standard jobs before the v0.1.0 tag is pushed
**Plans**: TBD
**Constraints**: ⚠️ Hard June 2, 2026 deadline — GitHub forces Node.js 24 default; CI workflow must be updated before that date. Do NOT start Phase 19, 20, or 21 before this phase closes.

### Phase 19: Example Phoenix App
**Goal**: A developer can clone the repo, run one command, and see Cairnloop's draft/approval/KB flow running in the browser — with documentation that explains how to add cairnloop to their own app
**Depends on**: Phase 18 (DEMO-04 requires `{:cairnloop, "~> 0.1"}` published to hex.pm — not a path dep)
**Requirements**: DEMO-01, DEMO-02, DEMO-03, DEMO-04
**Success Criteria** (what must be TRUE):
  1. Running `mix setup` then `mix phx.server` inside `examples/cairnloop_example/` boots the app with seeded data — no manual DB steps
  2. An operator visiting the running app can see a conversation go through draft → approval → KB flow in the browser without writing any code
  3. The example app's README explains how to add `{:cairnloop, "~> 0.1"}` to a host app and wire up required configuration
  4. CI `mix deps.tree` confirms the example app references the published hex dep, not a path dependency

### Phase 20: MCP OAuth Seam
**Goal**: MCP clients must present a valid OAuth 2.1 Bearer token to reach write endpoints; the token lifecycle is Ecto-backed and host-controlled
**Depends on**: Phase 18 (MCP protocol version bump from "2025-03-26" to "2025-11-05" ships with ExDoc metadata in the release)
**Requirements**: MCP-02, MCP-03
**Success Criteria** (what must be TRUE):
  1. An MCP client presenting a valid Bearer token can call `tools/list` and receives a 200 response
  2. An MCP client with an absent or invalid Bearer token on a write-capable route receives 401 with a `WWW-Authenticate` header containing an RFC 9728 resource-metadata pointer
  3. The `/.well-known/oauth-protected-resource` endpoint is served and returns the resource metadata document per RFC 9728
  4. Token issue, validation, and revocation operate against Ecto records storing the SHA-256 hash — the raw token is never persisted
**Plans**: TBD
**Constraints**: Cairnloop is a resource server only — it validates tokens, never issues them as an authorization server. Never call `Tool.run/3` from MCP handler code. MCP protocol version must be bumped from "2025-03-26" to "2025-11-05" in this phase.

---

### Phase 21: MCP Write Tools
**Goal**: An authenticated MCP client can invoke write-capable governed tools via `tools/call` and receives a `proposal_id` + `pending_approval` status — the full governed-action approval flow applies, not a shortcut
**Depends on**: Phase 20 (ACT-02 handler reads `conn.private` values set by the Auth Plug added in Phase 20)
**Requirements**: ACT-02, ACT-03
**Success Criteria** (what must be TRUE):
  1. An authenticated MCP client calling `tools/call` with a valid governed-tool name and input receives a response containing `proposal_id` and `"pending_approval"` status
  2. Every `tools/call` invocation creates a `ToolProposal` via `Governance.propose/3` — no direct call to `Tool.run/3` occurs anywhere in the MCP handler path
  3. A duplicate `tools/call` within the approval window (same tool, same idempotency key, same actor) returns the existing proposal rather than creating a new one
  4. An unauthenticated or unauthorized `tools/call` request is rejected at the Auth Plug before reaching `Governance.propose/3`
**Plans**: TBD
**Constraints**: `Tool.run/3` must NEVER be called from the MCP handler — this is a hard architectural constraint from the governed-action contract. One-active-lane idempotency must be extended to MCP origination.

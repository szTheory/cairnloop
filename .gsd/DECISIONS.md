- **M002-S01 Data Model**: Drafts (`cairnloop_drafts`) will explicitly `belong_to` a specific inbound `Message` (`in_reply_to_message_id`) while retaining `conversation_id` for query ergonomics. This ensures strict AI governance, enables context drift/collision detection, and supports threaded replies.
- **M002-S01 UI Seam**: The AI Draft "Audit Cockpit" will live in a **Dedicated Right Sidebar Panel**. This provides infinite vertical real estate for governance metadata (citations, policy gates), prevents context drift by keeping the main timeline clear, and avoids LiveView complexities of floating widgets. Operators will use an "Apply to Composer" action to move the draft for final editing.
- **M002-S01 Draft Lifecycle**: Drafts are immutable and append-only. When an operator interacts with a draft (Edits, Discards, or Approves), the system updates the `status` field (e.g., `:discarded`, `:approved`) rather than hard-deleting the record. This is mandatory to feed the Scoria evaluation flywheel with explicit rejection/approval data.
- **M002-S01 Collision Handling**: If a new inbound message arrives while a draft is in the `:proposed` state, the backend will automatically transition the draft's status to `:stale`. The UI will instantly remove it from the Audit Cockpit to prevent the operator from accidentally sending a response that ignores the customer's newest context.
- **M002-S01 Citations Structure**: Citations will be stored as a structured `jsonb` array (e.g. `citations: [%{url: "...", title: "..."}]`) on the Draft schema, rather than just embedded as markdown links in the body. This allows the LiveView sidebar to render them as distinct, clickable evidence cards.
- **M004-S01 Actor Validation**: `resolve_conversation` will strictly pattern match the `resolved_by` actor map (e.g., `%{type: _, id: _}`) to prevent the metadata from becoming an unstructured dumping ground, enforcing provenance for all resolutions.
- **M004-S01 Notifier Signature**: The `Cairnloop.Notifier.on_conversation_resolved` callback signature will be updated to explicitly include the `actor` argument (e.g., `on_conversation_resolved(conversation, actor, metadata)`). This surfaces the most critical piece of resolution context directly in the behaviour contract.
- **M004-S01 Reopen Telemetry**: Reopening a conversation (e.g., via `reply_to_conversation`) will clear the `resolved_at` timestamp and must emit a `[:cairnloop, :conversation, :reopened]` telemetry event. This is required to track reopen rates as a reliability signal for Parapet/SLOs.
- **Phase 1 Telemetry Architecture**: Implement "Dual Emission" for conversation resolution. `Cairnloop.Telemetry.span` with `[:cairnloop, :conversation, :resolve]` remains for internal Ecto transaction tracing/APM, but the codebase will explicitly execute `[:cairnloop, :conversation, :resolved]` as a pristine, past-tense domain event specifically for host application business logic (e.g., Webhooks, CRMs).
- **M005-S01 Threadline Durability**: Adopt the `Cairnloop.Auditor` behavior injected via `Ecto.Multi`. This ensures compliance-grade auditing by committing audit records in the exact same database transaction as the operator action, guaranteeing durable evidence.
- **M005-S02 Metrics Emission**: Emit strictly partitioned Telemetry metadata (e.g., `labels` and `evidence`) and distribute a generated Parapet Instrumenter into the host app via Igniter. This ensures high-cardinality safety and aligns with Parapet's operator-owned metrics tenet.
- **M005-S03 Reliability Artifacts**: Distribute SLO definitions (`Parapet.SLO.define`), runbooks, and `mix parapet.doctor` checks exclusively via `Igniter` code generation. No hidden macros, ensuring operators have perfect visibility and control.
